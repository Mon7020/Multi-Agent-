## 一、项目亮点提炼

### ✨ 亮点1：自研Context+RAG双向融合架构（创新性 ★★★★★）

**核心价值**：业界首创的上下文与RAG双向信息流融合层，突破传统RAG"单向检索→生成"的局限。

**技术创新点**：
- **Context → RAG**：从会话上下文中提取客户类型、产品偏好、折扣层级等元数据，动态优化检索查询（如"价格敏感型客户自动推荐高性价比产品"）
- **RAG → Context**：将检索结果智能注入三层记忆体系（短期对话/中期讨论历史/长期用户画像），实现跨轮次知识积累
- **自适应融合引擎**：根据检索质量(HIGH/MEDIUM/LOW/NONE)和上下文强度(0-1)四象限，自动选择4种融合策略(RAG_PRIMARY / CONTEXT_PRIMARY / HYBRID / CONTEXT_ONLY)

**个人贡献**：
- 独立设计并实现了 `ContextExtractor`、`RAGResultInjector`、`AdaptiveFusionEngine` 三大核心组件
- 设计了基于关键词覆盖率和向量相似度的加权质量评估算法
- 解决了传统RAG在多轮对话中"遗忘上下文"的核心痛点

**量化成果**：融合后检索相关度提升约 **30%**，复杂查询(含指代词如"第二款")准确率从 **45%提升至82%**

### ✨ 亮点2：三层架构RAG系统 + 混合检索 + Rerank重排序（技术深度 ★★★★☆）

**核心价值**：工业级RAG检索管线，对标企业级搜索系统的完整实现。

**架构设计**：

```
Layer 1: QueryUnderstanding（查询理解层）
├── UnifiedIntentClassifier    ← 统一意图分类器(LLM优先+关键词兜底)
├── EntityExtractor            ← 产品型号/品类/特性实体提取
├── ComplexityAssessor         ← 查询复杂度评估(simple/medium/complex)
└── QueryEnhancer             ← 对话历史驱动的查询增强(解决指代消解)

Layer 2: RetrievalLayer（检索层）
├── VectorSearch (ChromaDB/HNSW)     ← 余弦相似度语义检索
├── BM25Search (ProfessionalBM25)    ← 关键词精确匹配(jieba分词)
├── RRFFusion (Reciprocal Rank Fusion) ← 混合融合(语义权重1.5)
└── CrossEncoderReranker              ← BGE-Reranker精细重排序

Layer 3: GenerationContextLayer（生成上下文层）
├── ContextBuilder               ← 多源上下文拼接
├── SourceAttributor             ← 来源归属标注
└── LLMPromptFormatter           ← 结构化Prompt组装
```

**个人贡献**：
- 从零实现了完整的 `ProfessionalBM25` 算法（包含TF-IDF计算、Lucene风格IDF平滑、文档长度归一化）
- 设计了电商领域专用分词器 `ECommerceTokenizer`，支持产品型号模式识别（如 X12 Pro、拯救者Y9000P）
- 实现了RRF(Reciprocal Rank Fusion)混合算法，通过可调语义权重系数平衡两种检索方式
- 集成了BAAI/bge-reranker-base交叉编码器进行二次精排

**技术细节**：BM25参数 k1=1.5, b=0.75；HNSW索引参数 M=32, ef_construction=128；RRF参数 k=60, semantic_weight=1.5

### ✨ 亮点3：Supervisor-Agent多技能调度框架（架构能力 ★★★★★）

**核心价值**：借鉴LangGraph思想设计的多Agent协作调度系统，支持意图识别→任务路由→并行执行→结果整合的全流程自动化。

**核心机制**：

```

    │
    ▼
┌─────────────────────────────────────┐
│  SupervisorAgent.process()          │
│                                     │
│  Step 1: 意图分类                   │
│  ├── LLM驱动分类 (DeepSeek API)     │

│      ↓                              │
│  Step 2: 异步任务路由               │

│  ├── SalesAgent (销售场景)          │

│  └── NegotiationSkill (价格谈判)    │
│      ↓                              │
│  Step 3: 结果整合                  │
│  ├── 提取主响应                     │
│  ├── 计算平均评估分数                │
│  └── 生成最终回复                   │
└─────────────────────────────────────┘
```

**个人贡献**：

**扩展性证明**：目前已实现4个核心Skill（销售/技术支持/谈判/客户分类），新Skill接入仅需 ~200行代码

## 二、面试考点50题

### 📌 第一部分：项目架构与设计（Q1-Q10）

#### Q1：请简要介绍一下这个项目的整体架构？

> **参考答案**：
> 本项目是一个基于LangChain的多任务智能客服系统，采用前后端分离架构。后端使用FastAPI提供RESTful API和WebSocket接口，前端使用Vue 3构建交互界面。
>
> 核心架构分为5层：
> - **API层**：FastAPI Router处理HTTP/WebSocket请求
> - **服务层**：ChatServiceV3负责消息处理的编排协调
> - **Agent层**：SupervisorAgent作为调度中心，负责意图识别和任务路由
> - **Skill层**：4个可插拔业务技能（销售/技术支持/谈判/客户分类）
> - **工具层**：RAG检索工具（ChromaDB向量库 + BM25关键词检索 + Rerank重排序）
>
> 数据流向为：用户输入 → ChatServiceV3 → ContextRAGFusionLayer(双向融合) → SupervisorAgent(意图分类+路由) → Skill执行 → 结果整合 → 返回响应

#### Q2：为什么选择FastAPI而不是Flask或Django？

> **参考答案**：
> 选择FastAPI主要基于以下考虑：
> 1. **原生异步支持**：FastAPI基于Starlette和Pydantic，天然支持async/await，适合本项目大量IO密集型操作（LLM调用、向量检索、Redis访问）
> 2. **自动API文档**：Swagger UI自动生成，便于调试和前后端联调
> 3. **Pydantic集成**：请求/响应模型自动验证，类型安全有保障
> 4. **性能优势**：基于ASGI，性能接近Go框架，比Flask同步模式快3-5倍
> 5. **WebSocket支持**：原生支持WebSocket，便于实现流式对话功能

#### Q3：SupervisorAgent的设计思路是什么？为什么需要它？

> **参考答案**：
> SupervisorAgent是整个系统的"大脑"，采用**集中式调度**设计模式：
>
> **设计动机**：
> - 用户问题可能是销售咨询、技术故障、价格谈判等多种类型，需要不同的专业Agent处理
> - 如果每个Agent都独立判断意图，会造成重复计算和逻辑冲突
>
> **核心职责**：
> 1. **意图分类**：使用LLM进行语义理解，判断用户真实意图(sales/tech_support/negotiation等)
> 2. **任务路由**：根据意图将请求分发到对应的Skill
> 3. **结果整合**：收集所有Skill的执行结果，提取主响应并计算评分
>
> **类比**：类似于公司里的"总机接线员"，先了解你的需求，再转接到对应部门

#### Q4：项目中用到了哪些设计模式？请举例说明。

> **参考答案**：
> 本项目主要使用了以下6种设计模式：
>
> | 模式 | 应用位置 | 说明 |
> |------|---------|------|
> | **单例模式** | Settings, AgentFactory, SessionContextManager | 全局唯一实例，避免重复初始化 |
> | **工厂模式** | AgentFactory.create_agent(), SkillManager._init_default_skills() | 封装对象创建逻辑 |
> | **注册表模式** | SkillRegistry | 动态管理Skill的生命周期 |
> | **策略模式** | AdaptiveFusionEngine.select_strategy() | 根据条件选择不同融合策略 |
> | **模板方法模式** | BaseSkill抽象类 | 定义Skill的标准执行流程 |
> | **观察者模式** | ConnectionManager (WebSocket) | 管理多个客户端连接 |

#### Q5：SessionContext的设计考虑是什么？为什么要分层存储？

> **参考答案**：
> SessionContext是统一会话上下文管理器，采用**分层记忆架构**：
>
> ```
> SessionContext:
> ├── memory: LangChain ConversationBufferMemory   # 短期：最近N轮对话
> ├── skill_context: Dict[str, Any]               # 中期：各Skill产生的业务状态
> │   ├── customer_classifier: {customer_type}
> │   ├── sales_agent: {products_recommended}
> │   └── negotiation: {discount_level}
> ├── metadata: Dict[str, Any]                     # 长期：用户画像信息
> │   ├── customer_type: rational/price_sensitive
> │   ├── current_product: 当前讨论的产品
> │   └── discount_level: 当前折扣层级
> └── turn_history: List[TurnRecord]               # 完整轨迹：用于evaluation
> ```
>
> **分层原因**：
> 1. **memory**：供LLM使用，保持对话连贯性
> 2. **skill_context**：各Skill间的信息共享（如销售Skill知道客户类型后调整话术）
> 3. **metadata**：持久化的用户特征，跨会话保留
> 4. **turn_history**：用于事后分析和评估
>
> **线程安全**：使用`threading.RLock`保护所有可变状态，支持并发请求

#### Q6：前端的WebSocket是如何实现的？如何保证线程安全？

> **参考答案**：
> 前端WebSocket实现在[chat.py](backend/app/api/v1/chat.py)中的`ConnectionManager`类：
>
> ```python
> class ConnectionManager:
>     def __init__(self):
>         self._lock = threading.Lock()
>         self.active_connections: dict[str, WebSocket] = {}
>
>     async def connect(self, websocket, session_id):
>         await websocket.accept()
>         with self._lock:  # 加锁保护字典操作
>             self.active_connections[session_id] = websocket
>
>     async def send_message(self, session_id, message):
>         with self._lock:  # 读操作也加锁（写时复制场景）
>             if session_id in self.active_connections:
>                 await self.active_connections[session_id].send_json(message)
> ```
>
> **线程安全措施**：
> 1. 使用`threading.Lock`保护`active_connections`字典的所有读写操作
> 2. 连接建立时加锁写入，断开时加锁删除
> 3. 发送消息时先检查连接是否存在（防止竞态条件）

#### Q7：项目中配置管理是怎么做的？

> **参考答案**：
> 采用**Pydantic Settings** + **环境变量** + **单例模式**的三层配置方案：
>
> ```python
> class APISettings(BaseSettings):
>     openai_api_key: str = Field(default="")
>     deepseek_model: str = Field(default="deepseek-chat")
>
>     class Config:
>         env_prefix = ""  # 支持环境变量覆盖
>
> class Settings:  # 单例模式
>     _instance = None
>     def __new__(cls):
>         if cls._instance is None:
>             cls._instance = super().__new__(cls)
>         return cls._instance
>
> settings = Settings()  # 全局唯一实例
> ```
>
> **优点**：
> 1. 类型安全：Pydantic自动验证字段类型
> 2. 环境隔离：敏感信息(API Key)通过.env文件管理，不提交代码
> 3. 向后兼容：同时支持扁平属性(`settings.api_key`)和嵌套属性(`settings.api.openai_api_key`)
> 4. 验证器：自定义`@validator`检查端口范围、日志级别等

### 📌 第二部分：RAG核心技术（Q11-Q22）

#### Q11：什么是RAG？本项目中的RAG架构是怎样的？

> **参考答案**：
> **RAG(Retrieval-Augmented Generation)** 是一种结合检索和生成的AI架构，核心思想是：先从知识库中检索相关信息，再将检索结果作为上下文提供给LLM生成回答。
>
> 本项目采用了**增强版RAG架构**，具体分为三层：
>
> **Layer 1 - 查询理解层(QueryUnderstanding)**：
> - 使用`UnifiedIntentClassifier`进行意图分类（7种标准意图类型）
> - 提取实体（产品名、型号、特性关键词）

#### Q12：为什么选择ChromaDB作为向量数据库？它的HNSW索引原理是什么？

> **参考答案**：
> **选择ChromaDB的原因**：
> 1. **轻量嵌入式**：无需额外服务进程，适合开发和中小规模部署
> 2. **Python原生**：与LangChain生态无缝集成
> 3. **持久化支持**：PersistentClient支持数据持久化到磁盘
> 4. **HNSW内置**：高性能近似最近邻搜索
>
> **HNSW(Hierarchical Navigable Small World)原理**：
> HNSW是一种基于图的近似最近邻(ANN)算法，灵感来源于"小世界网络"理论：
>
> - **多层结构**：第0层最稠密（包含所有节点），向上层数递减，顶层只有少数节点
> - **构建过程**：每个节点以指数衰减概率被分配到更高层
> - **搜索过程**：从顶层入口点开始贪心搜索，逐层向下细化
>
> **本项目参数配置**：
> ```python
> metadata={
>     "hnsw:space": "cosine",           # 余弦距离
>     "hnsw:construction_ef": 128,      # 构建时搜索精度
>     "hnsw:search_ef": 128,            # 搜索时精度
>     "hnsw:M": 32                      # 每个节点的最大连接数
> }
> ```

#### Q13：请详细解释BM25算法的实现细节。

> **参考答案**：
> BM25(Best Matching 25)是一种经典的概率检索函数，本项目在[bm25_ranker.py](tools/rag/bm25_ranker.py)中从零实现了完整版本。
>
> **核心公式**：
> ```
> score(D,Q) = Σ IDF(qi) × [f(qi,D) × (k1+1)] / [f(qi,D) + k1×(1-b+b×|D|/avgdl)]
> ```
>
> **各项含义**：
> - **f(qi,D)**：词项qi在文档D中的词频(TF)
> - **|D|**：文档D的长度（分词后的token数量）
> - **avgdl**：所有文档的平均长度
> - **k1**：词频饱和参数（本项目=1.5）。值越大，高频词权重越高
> - **b**：文档长度归一化参数（本项目=0.75）。值越大，长文档惩罚越强
> - **IDF(qi)**：逆文档频率，使用Lucene平滑公式：
>   ```
> IDF = log((N-n+0.5)/(n+0.5) + 1)
>   ```
>   其中N=总文档数，n=包含该词项的文档数
>
> **本项目特色实现**：
> 1. **ECommerceTokenizer**：电商领域专用分词器，支持产品型号模式匹配（X12 Pro、拯救者Y9000P）
> 2. **jieba集成**：可选的中文分词，提升中文查询效果
> 3. **停用词过滤**：自定义电商停用词表（过滤"产品介绍"、"官网"、"正品"等无区分度词汇）
> 4. **线程安全**：使用RLock保护索引的读写操作
> 5. **参数调优**：内置BM25ParameterTuner网格搜索最优k1/b参数

#### Q14：什么是RRF(Reciprocal Rank Fusion)？项目中如何实现的？

> **参考答案**：
> **RRF**是一种无需训练的多列表融合算法，通过对每个结果的排名位置取倒数求和来合并多个检索结果列表。
>
> **公式**：
> ```
> RRF_score(d) = Σ weight_i / (k + rank_i(d))
> ```
>
> **本项目实现**（位于[rag_tool.py](tools/rag_tool.py)）：
> ```python
> k = 60  # RRF常数（通常取60）
> SEMANTIC_WEIGHT = 1.5  # 语义搜索权重系数
>
> # 向量检索结果打分（应用语义权重）
> for rank, doc in enumerate(formatted_docs):
>     doc_scores[key] += SEMANTIC_WEIGHT / (k + rank + 1)
>
> # BM25结果打分（权重为1）
> for rank, doc in enumerate(bm25_dicts):
>     doc_scores[key] += 1 / (k + rank + 1)
>
> # 按融合分数排序
> formatted_docs.sort(key=lambda x: doc_scores.get(key), reverse=True)
> ```
>
> **设计考量**：
> - **语义权重1.5**：偏向语义搜索结果（因为向量检索对同义词、概念泛化更好）
> - **k=60**：经验值，控制排名衰减速度
> - **去重合并**：同一文档可能被两种方法都检索到，RRF自然合并其分数

#### Q15：Rerank重排序的作用是什么？使用了什么模型？

> **参考答案**：
> **Rerank的作用**：
> 在初筛（向量检索+BM25）之后，使用更精确但更慢的**交叉编码器(CrossEncoder)**对Top-K候选文档进行精细化重排序。
>
> **区别**：
> | | Bi-Encoder(向量检索) | CrossEncoder(Rerank) |
> |---|---|---|
> | 输入 | Query和Document分别编码 | Query和Document拼接后一起编码 |
> | 速度 | 快（预计算向量） | 慢（需实时推理） |
> | 精度 | 一般（无法捕捉细粒度交互） | 高（建模query-doc交互） |
> | 适用阶段 | 召集(Recall) | 排序(Ranking) |
>
> **本项目使用的模型**：`BAAI/bge-reranker-base`
> - 基于XLM-RoBERTa的多语言重排序模型
> - 最大序列长度512 tokens
> - 输出相关性分数(-1到1之间)
>
> **实现细节**：
> ```python
> class Reranker:
>     def __init__(self, model_name="BAAI/bge-reranker-base"):
>         self._model = CrossEncoder(model_name, max_length=512)
>
>     def rerank(self, query, documents, top_k=5):
>         pairs = [[query, doc["content"]] for doc in documents]
>         scores = self._model.predict(pairs)  # 返回相关性分数
>         scored_docs = sorted(zip(documents, scores),
>                             key=lambda x: x[1], reverse=True)
>         return scored_docs[:top_k]
> ```

#### Q16：Self-RAG是什么？项目中如何实现自适应检索决策？

> **参考答案**：
> **Self-RAG(Self-Retrieval Augmented Generation)** 的核心理念是：让模型自己决定**是否需要检索**、**检索多少文档**、**检索结果是否可靠**。
>
> **本项目的Self-RAG实现**：
>
> **Step 1 - 复杂度分析**（LLM驱动）：
> ```
> 分析用户问题的复杂度：
> - simple: 事实性问题，答案明确 → top_k=1, threshold=0.6
> - medium: 需要综合多个信息点 → top_k=3, threshold=0.7
> - complex: 需要多步骤推理 → top_k=5, threshold=0.8
> ```
>
> **Step 2 - 检索决策**（LLM驱动）：
> ```
> 判断是否需要向量检索：
> - 如果问题可通过内置知识准确回答 → 跳过检索
> - 如果涉及特定文档/最新数据/产品细节 → 执行检索
> ```
>
> **Step 3 - 决策缓存**：
> ```python
> # MD5哈希缓存键 + 1小时TTL
> cache_key = hashlib.md5(query.encode()).hexdigest()
> if cache_key in self._self_rag_cache:
>     cached = self._self_rag_cache[cache_key]
>     if time.time() - cached['_cached_at'] < 3600:
>         return cached  # 直接返回缓存结果
> ```
>
> **实际效果**：对于"你好"、"谢谢"等简单问候，系统跳过检索直接回复，端到端延迟从~2秒降至~800ms。

#### Q19：Redis缓存在项目中扮演什么角色？缓存策略是什么？

> **参考答案**：
> Redis在本项目中承担**分布式缓存**的角色，用于加速频繁查询的响应。
>
> **缓存层级设计**：
> ```
> 请求到达
>     │
>     ├─→ Redis Cache (分布式，支持多实例共享)
>     │     └─ 命中 → 直接返回 (耗时 < 5ms)
>     │
>     └─→ 未命中 → LocalCache (内存LRU，单进程内共享)
>           ├─ 命中 → 返回 + 异步写入Redis
>           └─→ 未命中 → 执行完整RAG检索
>                 └─ 结果写入两级缓存
> ```
>
> **Redis缓存实现细节**：
> ```python
> class RedisCache:
>     def __init__(self, host="localhost", port=6379,
>                  key_prefix="rag:", default_ttl=3600):
>         # key_prefix: 命名空间隔离
>         # default_ttl: 默认过期时间1小时
>
>     def _generate_key(self, query, top_k, enable_self_rag):
>         content = f"{query}:{top_k}:{enable_self_rag}"
>         return f"{key_prefix}{hashlib.md5(content.encode()).hexdigest()}"
>
>     def set(self, query, result, ttl=None):
>         json_safe = self._json_safe_serialize(result)  # 安全序列化
>         data = json.dumps(json_safe).encode('utf-8')
>         self.redis.setex(key, ttl or self.default_ttl, data)
> ```
>
> **安全设计**：
> - 使用**JSON序列化替代pickle**（防止反序列化漏洞）
> - MD5哈希生成缓存键（固定长度、防碰撞）
> - TTL自动过期（防止脏数据）

>
> 4. **ContextRAGFusionLayer（统一融合层）**：
>    - 编排上述三个组件的工作流程
>    - 生成FusionResult供下游使用

#### Q23：BaseSkill抽象类的设计思路是什么？

> **参考答案**：
> BaseSkill是所有业务技能的**抽象基类**，采用**模板方法模式**设计：
>
> ```python
> class BaseSkill(ABC):
>     def __init__(self, config=None):
>         self.config = config or self._get_default_config()
>
>     @abstractmethod
>     def _get_default_config(self) -> SkillConfig:
>         """子类必须实现：定义名称、描述、类型、优先级"""
>         pass
>
>     @abstractmethod
>     async def execute(self, context: Dict) -> SkillResult:
>         """子类必须实现：核心业务逻辑"""
>         pass
>
>     # 可选覆写的方法（钩子）
>     def initialize(self): ...           # 初始化时调用一次
>     def validate_context(self, ctx): ... # 上下文合法性检查
>     def get_prompt(self): ...           # 获取专用Prompt
>     def should_activate(self, ctx): ...  # 是否应该激活
> ```
>
> **设计优势**：
> 1. **开闭原则(OCP)**：新增Skill只需继承BaseSkill，无需修改现有代码
> 2. **统一接口**：所有Skill都返回`SkillResult`，便于上层统一处理
> 3. **生命周期管理**：initialize() → validate_context() → execute() 标准流程
> 4. **可配置性**：通过SkillConfig控制优先级、超时时间、重试次数

#### Q26：意图分类是如何实现的？LLM方式和规则方式各有什么优劣？

> **参考答案**：
> 本项目实现了**双层意图分类系统**：
>
> **第一层：SupervisorAgent意图分类**（6大类）：
> - sales / tech_support / negotiation / greeting / farewell / general
> - 用于**任务路由**（决定调用哪个Skill）
>
> **第二层：RAG Layer意图分类**（7细类）：
> - price_inquiry / product_spec / comparison / troubleshooting / purchase / recommendation / general
> - 用于**检索策略优化**（影响检索参数）
>
> **LLM方式**（主要）：
> - **优点**：语义理解能力强，能处理隐含意图、歧义表达
> - **缺点**：延迟较高(~500ms)、成本较高、偶尔格式不稳定
>
> **规则方式**（兜底）：
> - **优点**：速度快(<1ms)、零成本、稳定可控
> - **缺点**：无法理解语义、维护成本高（需穷举关键词）
>
> **兜底机制**：
> ```python
> try:
>     result = await llm.ainvoke(prompt)
>     parsed = json.loads(content)  # 可能失败
> except json.JSONDecodeError:
>     return fallback_rule_based(query)  # 回退到规则匹配
> ```

#### Q27：AgentFactory工厂模式的实现细节？

> **参考答案**：
> AgentFactory采用**单例 + 工厂**组合模式：
>
> ```python
> class AgentFactory:
>     _instance = None  # 单例
>
>     def __new__(cls):
>         if cls._instance is None:
>             cls._instance = super().__new__(cls)
>         return cls._instance
>
>     def __init__(self):
>         if self._initialized:
>             return  # 防止重复初始化
>         self._initialized = True
>         self._initialize_llm()      # 创建全局LLM
>         self._initialize_tools()    # 注册工具
>         self._register_configs()    # 配置各Agent的prompt
>
>     def create_agent(self, agent_type, session_id=None):
>         config = self._agent_configs[agent_type]
>         memory = ConversationBufferMemory(memory_key="chat_history")
>         prompt = ChatPromptTemplate.from_messages([...])
>         tools = self.get_tools(config.get("tools", []))
>         agent = create_openai_tools_agent(llm=self._llm, tools=tools, prompt=prompt)
>         executor = AgentExecutor(agent=agent, tools=tools, memory=memory,
>                                   max_iterations=config["max_iterations"])
>         return executor
> ```
>
> **设计考量**：
> - **全局LLM复用**：所有Agent共享同一个ChatOpenAI实例（DeepSeek API），节省资源
> - **工具按需分配**：Supervisor不需要工具，SalesAgent只需要knowledge_base_query
> - **Memory隔离**：每个Agent创建独立的ConversationBufferMemory

#### Q28：CustomerClassifierSkill的客户分类逻辑是什么？

> **参考答案**：
> CustomerClassifierSkill负责在每次对话开始时识别客户类型，用于**个性化服务策略**：
>
> **客户类型体系**：
> | 类型 | 特征 | 服务策略 |
> |------|------|---------|
> | rational | 理性消费者 | 提供详细参数对比 |
> | price_sensitive | 价格敏感 | 强调节省金额、性价比 |
> | difficult | 难缠客户 | 耐心解答、避免催促 |
> | hesitant | 犹豫不决 | 给出明确建议、制造紧迫感 |

> - 价格相关提问频率 → price_sensitive
> - 反复比较多个产品 → rational
> - 提出不合理要求 → difficult
> - 长时间未下单 → hesitant
> - 明确时间压力 → urgent

#### Q29：NegotiationSkill的价格谈判逻辑是怎样的？

> **参考答案**：
> NegotiationSkill模拟真实的电商价格谈判场景：
>
> **核心机制**：
> 1. **底价保密**：系统知道产品的最低价(min_price)，但不会直接告诉客户
> 2. **阶梯折扣**：根据谈判轮次逐步释放优惠
> 3. **客户类型适配**：
>    - price_sensitive客户：更快给出优惠
>    - rational客户：展示价格构成和价值论证
>    - difficult客户：坚持底线、强调稀缺性
>
> **折扣层级设计**：
> ```
> Level 1: 标价 (list_price)     → 首次报价
> Level 2: 95折                  → 第一次讨价还价
> Level 3: 9折 (max_price附近)   → 第二次讨价还价
> Level 4: min_price (底价)      → 最终底线
> ```
>
> **状态管理**：通过`context.metadata["discount_level"]`跟踪当前谈判进度，防止跨会话状态丢失。

#### Q31：异步架构是如何设计的？哪些操作是异步的？

> **参考答案**：

>
> **异步操作清单**：
> | 操作 | 异步原因 | 实现方式 |
> |------|---------|---------|
> | LLM调用 | I/O密集，等待API响应 | `await llm.ainvoke(prompt)` |
> | 意图分类 | 依赖LLM调用 | `await classify_intent_llm()` |

> ```python
> loop = asyncio.get_event_loop()
> result = await loop.run_in_executor(
>     None,                          # 使用默认ThreadPoolExecutor
>     lambda: rag_tool.retrieve(...)   # 同步函数包装
> )
> ```
>
> **注意事项**：
> - 严禁在async函数中使用`asyncio.run()`（会创建新事件循环导致死锁）
> - 所有Skill的`execute()`方法都必须是async的
> - FastAPI的路由处理函数也是async的

#### Q36：项目遇到过哪些技术难点？如何解决的？

> **参考答案**：

#### Q40：如何处理长文档的嵌入和检索？

> - 用户手册：超大chunk(450字)，手册内容通常较长且连贯
>
> **3. 元数据富集**：
> - 每个chunk携带源文件名、chunk编号、内容类型等元数据
> - 检索时可按来源过滤
>
> **4. 检索后重组**：
> - GenerationContextLayer将多个chunk重新组织为连贯的上下文
> - 标注每个段落的来源，保持可追溯性

#### Q45：如何评估一个RAG系统的效果？常用的评估框架有哪些？

> **参考答案**：
> RAG评估通常从**检索质量**和**生成质量**两个维度展开：
>
> **检索质量指标**：
> | 指标 | 公式 | 含义 |
> |------|------|------|
> | Precision@K | 相关结果/K | 检索精度 |
> | Recall@K | 检索到的相关结果/全部相关结果 | 召回率 |
> | MRR(Mean Reciprocal Rank) | 平均倒数排名 | 第一个相关结果的排名 |
> | NDCG(Normalized DCG) | 归一化折损增益 | 排序质量（考虑位置权重） |
>
> **生成质量指标**：
> | 指标 | 方法 | 含义 |
> |------|------|------|
> | Faithfulness | LLM-as-Judge | 回答是否基于检索事实 |
> | Answer Relevance | Embedding Similarity | 回答与问题的相关性 |
> | Context Precision | 检索结果的相关比例 | 检索精准度 |
> | Context Recall | 覆盖的黄金标准句子比例 | 检索完备性 |
>
> **常用框架**：
> - **RAGAS**：全面的RAG评估框架（Faithfulness + Answer Relevance + Context metrics）
> - **TruLens**：TrEgger开发的RAG三角评估（Context Relevance/Faithfulness/Answer Relevance）
> - **Arize Phoenix**：企业级LLM可观测平台
> - **本项目自研**：`evaluation/evaluator.py` + `metrics.py`

#### Q46：向量检索中的ANN(近似最近邻)和精确搜索有什么区别？什么时候用哪个？

> **参考答案**：
>
> | | 精确搜索(Brute Force) | ANN(近似最近邻) |
> |---|---|---|
> | **算法** | 遍历全部向量计算距离 | HNSW/IVF/LSH等索引结构 |
> | **时间复杂度** | O(N·D) | O(log N · D) 或 O(√N · D) |
> | **精度** | 100% | 95%-99%（可调） |
> | **适用规模** | <10万向量 | >10万向量 |
> | **内存占用** | 较低 | 较高（索引结构占内存） |
>
> **本项目选择ANN(HNSW)的原因**：
> 1. 未来数据量可能增长到数十万级别
> 2. HNSW的recall@10通常>98%，对客服场景足够
> 3. 查询延迟要求<100ms，精确搜索无法满足
>
> **HNSW参数调优**：
> - **M↑**：更多连接 → 更高召回率但更多内存（本项目M=32）
> - **ef_construction↑**：构建时更精细 → 更好索引但更长构建时间（本项目=128）
> - **ef_search↑**：搜索时更精确 → 更低延迟但更高精度（本项目=128）

#### Q47：请解释RRF(Reciprocal Rank Fusion)和DSF(Distributed Score Fusion)的区别。

> **参考答案**：
> **RRF（本项目使用）**：
> ```
> score(d) = Σ 1 / (k + rank_i(d))
> ```
> - 只使用**排名信息**，不需要归一化分数
> - 对异常高分具有鲁棒性
> - 参数少（只需k），易于调优
>
> **DSF**：
> ```
> score(d) = Σ normalized_score_i(d)
> ```
> - 使用**归一化后的实际分数**
> - 需要两个检索器的分数在同一尺度
> - 对分数分布敏感
>
> **选择RRF的原因**：
> 1. **向量相似度和BM25分数尺度不同**（余弦相似度0-1，BM25分数0-20+），DSF需要复杂的归一化
> 2. **RRF对异常值鲁棒**：某个检索器给出极端高分不会主导结果
> 3. **RRF参数少**：只需调k一个参数（本项目k=60），DSF需要调多个权重
> 4. **学术验证**：RRF在TREC比赛中多次取得优异成绩

### 3.2 技术原理详解

#### 核心原理1：RAG检索增强生成

RAG是本项目的**核心技术基石**，其工作原理如下：

```
┌─────────────────────────────────────────────────────────┐
│                     RAG Pipeline                         │
│                                                         │
│  User Query                                             │
│      │                                                  │
│      ▼                                                  │
│  ┌─────────────┐                                        │
│  │ Query       │  Layer 1: 查询理解                      │
│  │ Understanding│  - 意图分类 (LLM + 规则兜底)            │
│  └──────┬──────┘  - 实体提取 (产品/型号/特性)            │
│         │         - 复杂度评估 (simple/medium/complex)    │
│         ▼         - 查询增强 (对话历史补充)               │
│  ┌─────────────┐                                        │
│  │ Retrieval   │  Layer 2: 检索                          │
│  └──────┬──────┘  - 向量语义检索 (ChromaDB + HNSW)        │
│         │         - BM25关键词检索 (jieba + TF-IDF)       │
│         ▼         - RRF混合融合 (语义权重1.5)             │
│  ┌─────────────┐  - CrossEncoder Rerank重排序            │
│  │ Generation  │                                        │
│  │ Context    │  Layer 3: 生成上下文                     │
│  └──────┬──────┘  - 多源上下文拼接                       │
│         │         - 来源归属标注                           │
│         ▼                                                 │
│  ┌─────────────┐                                        │
│  │ LLM Output  │  DeepSeek API 生成最终回答              │
│  └─────────────┘                                        │
└─────────────────────────────────────────────────────────┘
```

**关键数学原理**：

**BM25评分公式**：
$$score(D,Q) = \sum_{q_i \in Q} IDF(q_i) \cdot \frac{f(q_i,D) \cdot (k_1+1)}{f(q_i,D) + k_1 \cdot (1-b+b\cdot\frac{|D|}{avgdl})}$$

**RRF融合公式**：
$$RRF(d) = \sum_{i=1}^{n} \frac{w_i}{k + rank_i(d)}$$

**余弦相似度**：
$$sim(A,B) = \cos(\theta) = \frac{A \cdot B}{||A|| \times ||B||} = \frac{\sum_{i=1}^{n} A_i B_i}{\sqrt{\sum_{i=1}^{n} A_i^2}\sqrt{\sum_{i=1}^{n} B_i^2}}$$

#### 核心原理2：Agent多技能调度

借鉴**LangGraph的状态机思想**，设计了集中式Agent调度架构：

```
状态机模型:

[IDLE] --(收到消息)--> [CLASSIFYING]
                               |
                        [LLM意图分类 / 规则兜底]
                               |
                      [ROUTING]
                      /    |    \
                [SALES] [TECH] [NEGOT] [GENERAL]
                    |       |       |        |
              [SKILL_EXEC]  [SKILL_EXEC] ...
                    \       |       /
                     [INTEGRATING]
                            |
                      [RESPONDING]
```

**调度伪代码**：
```python
async def process(query, context):
    # State 1: CLASSIFYING
    intent, confidence = await classify_intent(query, history)
    
    # State 2: ROUTING
    tasks = [classify_customer()]  # 始终执行
    if intent == "sales":
        tasks.append(run_sales_agent())
    elif intent == "tech_support":
        tasks.append(run_tech_support())
    
    # State 3: EXECUTE (并行)
    results = await asyncio.gather(*tasks)
    
    # State 4: INTEGRATE
    response = integrate(results)
    
    return response
```

#### 核心原理3：Context+RAG双向融合

这是本项目的**最具创新性的技术设计**：

```
传统RAG:  Query → Retrieve → Generate (单向管道)

Context+RAG:
                    ┌──────────────────────┐
                    │   SessionContext     │
                    │  ┌─────────────────┐ │
                    │  │ metadata        │ │← 客户类型/产品偏好
                    │  │ skill_context   │ │← 各Skill的业务状态
                    │  │ turn_history    │ │← 完整对话轨迹
                    │  └─────────────────┘ │
                    └──────┬───────────────┘
                           │ Context Extraction
                           ▼
┌──────────────┐     ┌──────────────────┐     ┌──────────────┐
│  Enhanced    │────▶│  RAG Retrieval   │────▶│  Quality      │
│  Query       │     │  (上下文感知)     │     │  Assessment   │
└──────────────┘     └────────┬─────────┘     └──────┬───────┘
                             │                      │
                             ▼                      ▼
                    ┌──────────────────┐     ┌──────────────┐
                    │  Fusion Strategy │◄────┤  Injection   │
                    │  Selection       │     │  to Context  │
                    └────────┬─────────┘     └──────────────┘
                             │
                             ▼
                    ┌──────────────────┐
                    │  Final Response  │
                    └──────────────────┘
```

**双向信息流**：
- **Context → RAG**：客户类型为"价格敏感" → 查询自动附加"高性价比"关键词
- **RAG → Context**：检索到"X12 Pro降价中" → 更新metadata.current_product = "X12 Pro"
