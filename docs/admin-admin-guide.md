# 管理员指南

## 适用对象

本指南面向 `admin` 和 `super_admin`，用于说明后台知识库、记忆、设置和账号管理的日常操作方式。`operator` 也可进入后台，但知识库页面为只读模式。

## 后台入口

- 开发地址：`http://localhost:5174/admin.html`
- 后台 API 文档：`http://localhost:8001/docs`

## 角色边界

### `super_admin`

- 可访问全部后台模块
- 可调整账号角色
- 可修改系统运行参数

### `admin`

- 可访问总览、记忆管理、知识库管理、系统设置、账号管理
- 可上传、替换、删除、恢复知识文档
- 可修改知识文档的发布状态、前台显隐、描述、标签和角色范围

### `operator`

- 可访问总览、记忆管理、知识库管理
- 知识库页只读，可查看状态和指标
- 不可上传、替换、删除、恢复，也不可保存知识库设置

## 模块说明

### 总览

用途：

- 确认当前登录角色
- 快速查看后台模块接入状态

### 记忆管理

用途：

- 搜索用户记忆数据
- 查看上下文快照和长期偏好
- 修正偏好
- 清理上下文或全部记忆

注意：

- 所有写操作都会记录到 `logs/admin_audit.jsonl`

### 知识库管理

当前后台知识库模块已经支持完整运营闭环：

- 上传新知识文件
- 查看文档描述、标签、校验值、分块数量等指标
- 修改文档元数据和前台可见策略
- 替换文档文件且保留稳定 `document_id`
- 查看版本历史
- 基于历史版本安全回滚
- 软删除到回收区
- 从回收区恢复

关键规则：

- 前台只会显示：未删除、已发布、前台可见、当前角色允许访问的文档
- 恢复后的文档默认是“未发布 + 前台隐藏”
- 回滚只处理内容版本，不自动改动发布、显隐、角色范围和删除状态
- `operator` 可以看状态和指标，但不能执行任何写操作

推荐操作顺序：

1. 上传或替换文件后，先检查 `chunk_count` 是否符合预期
2. 补充描述、标签和允许访问角色
3. 确认 `visible_to_frontend`
4. 最后切换到 `published=true`
5. 使用目标角色账号到前台验证是否可见

### 系统设置

用途：

- 查看当前权限模型
- 查看前台只读策略
- 更新 RAG 运行参数

建议流程：

1. 先记录当前参数值
2. 每次只调整一组参数
3. 保存后检查前台检索表现
4. 如结果异常，按原值回退

### 账号管理

用途：

- 查看用户列表
- 调整角色

注意：

- 当前角色修改接口仅 `super_admin` 可用

## 知识库日常操作

### 上传新文档

1. 进入“知识库管理”
2. 点击“上传知识文件”
3. 选择 `.txt`、`.pdf` 或 `.docx`
4. 上传成功后检查：
   - 文件名
   - `chunk_count`
   - 描述和标签
   - 前台显隐
   - 发布状态

默认行为：

- 新上传文档默认建议保持“未发布 + 前台隐藏”
- 需要人工确认后再开放给前台

### 修改元数据

可直接在右侧详情区修改：

- 描述
- 标签
- 发布状态
- 前台可见
- 允许访问角色

保存后会更新注册表，并写入审计日志。

### 替换文件

适用场景：

- 文件内容更新，但业务上仍视为同一份文档

规则：

- 替换后保留原 `document_id`
- 会重建分块并刷新 `chunk_count`
- 如果文档之前已发布，替换后仍保留发布状态

建议：

- 替换后重点检查 `chunk_count` 是否异常变化

### 版本历史与回滚

适用场景：

- 错误替换了文件内容
- 新版本分块结果明显异常，需要回退到已知稳定版本
- 需要审计某个文档最近几次创建、替换和回滚记录

查看方式：

1. 在知识库列表中选中文档
2. 在右侧详情区进入“版本历史”
3. 选择目标版本查看 `filename`、`description`、`tags`、`chunk_count`、`checksum`

回滚规则：

- 会回滚：文件内容、`filename`、`description`、`tags`
- 不会回滚：`published`、`visible_to_frontend`、`allowed_roles`、`deleted`
- 回滚成功后，系统会生成一个新的当前版本，而不是把指针直接切回旧版本

操作建议：

1. 先确认目标版本的 `chunk_count` 和 `checksum` 符合预期
2. 再执行回滚
3. 回滚完成后重新检查当前指标
4. 人工再次确认发布状态、前台显隐和允许角色是否仍符合当前运营策略

限制：

- 已删除文档不能直接回滚，必须先恢复
- `operator` 只能查看版本历史，不能执行回滚

### 删除文档

删除行为是软删除：

- 文件会移入 `data/knowledge/trash/`
- 前台立即不可见
- 向量分块会被清理
- 后台可在“已删除”视图继续查看

### 恢复文档

恢复后系统会：

- 把文件移回活动目录
- 重建向量分块
- 强制设置为 `published=false`
- 强制设置为 `visible_to_frontend=false`

恢复后需要管理员再次确认角色范围和发布策略。

## 审计日志

默认文件：

- `logs/admin_audit.jsonl`

知识库模块重点记录：

- 文档创建
- 文档替换
- 元数据修改
- 访问控制修改
- 软删除
- 恢复

排查建议：

1. 先确认接口返回码是否为 `401`、`403`、`404` 或 `409`
2. 再检查账号角色和文档状态
3. 最后查看 `admin_audit.jsonl` 是否记录了操作结果

## 每日检查清单

1. `http://localhost:8000/api/v1/health` 正常
2. `http://localhost:8001/health` 正常
3. 后台知识库状态与前台实际可见结果一致
4. 最近上传或替换的文档 `chunk_count` 正常
5. 最近恢复的文档仍保持“未发布 + 前台隐藏”

## 风险提示

- 不要在前台处理知识库发布与显隐
- 不要跳过恢复后的二次审核
- 调整角色范围后，必须使用目标角色做前台验证
- 当 `chunk_count` 明显异常时，不要直接发布，应先检查文件内容和分块结果
