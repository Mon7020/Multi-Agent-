# 后台管理系统总设计（第 1 阶段：主框架与权限底座）

## 1. 背景

当前项目已经具备以下基础能力：

- 用户前台：登录、对话、知识库、设置
- 基础认证：注册、登录、Token 鉴权、当前用户查询
- 记忆管理后台：独立管理端页面与独立管理 API

但系统仍存在明显结构问题：

- 后台只覆盖“记忆管理”，并未形成统一后台产品
- 设置与知识库管理仍停留在用户前台，不符合后台治理边界
- 账号体系只有基础认证，没有后台用户管理、角色分配、权限控制
- 后台缺少统一导航、统一视觉语言、统一接口规范、统一审计日志

本设计的目标是将现有零散能力整合为统一后台管理系统，并以“可扩展、不返工”的方式推进。考虑到需求覆盖 4 个独立子系统，本项目拆为 4 个子项目逐步实施。

## 2. 子项目拆分

### 子项目 1：后台主框架与权限底座

本阶段为当前设计文档的实施范围，包括：

- 后台统一布局与导航
- 后台访问控制与角色模型
- 统一后台 API 边界
- 用户账号管理基础能力
- 记忆管理接入统一后台框架
- 知识库管理和系统设置的后台骨架与权限入口
- 审计日志、错误处理、数据验证的底座能力

### 子项目 2：记忆管理子模块

- 记忆数据的增删改查
- 分类管理
- 搜索筛选
- 批量操作
- 结构化展示和操作历史

### 子项目 3：知识库管理与系统设置子模块

- 知识库后台迁移
- 草稿 / 已发布双态版本模型
- 角色可见范围控制
- 系统参数配置
- 用户偏好设置
- 权限相关设置界面

### 子项目 4：用户账号管理增强

- 管理员密码重置
- 角色分配完善
- 权限策略管理
- 审计查询
- 更完整的账号生命周期管理

## 3. 本阶段目标

第 1 阶段的目标不是把所有后台功能一次性做完，而是完成“统一后台产品底座”，确保后续 3 个子项目接入时不需要重做：

- 后台入口统一
- 模块导航统一
- 权限模型统一
- API 风格统一
- 审计与安全边界统一
- 页面视觉语言统一

## 4. 用户与角色模型

后台复用现有账号体系，不单独维护一套管理员账号库。登录后通过用户角色决定是否能访问后台。

### 角色定义

- `super_admin`
  - 拥有全部后台权限
  - 可管理角色、权限、安全配置、模块可见性
- `admin`
  - 可管理记忆、知识库、用户、常规系统设置
  - 不能修改最高级安全策略
- `operator`
  - 可查看后台与执行低风险运营操作
  - 不可进行高风险删除、角色分配、核心配置修改
- `user`
  - 仅能访问前台
  - 无后台访问权限

### 权限原则

- 后台前端菜单与按钮按角色裁剪
- 所有后台 API 必须在服务端做最终权限校验
- 前端隐藏不视为安全措施
- 高风险操作必须写审计日志

## 5. 信息架构

后台统一为独立应用入口，采用单独的后台布局与导航体系。

### 顶层模块

- `Dashboard` 总览
- `记忆管理`
- `知识库管理`
- `系统设置`
- `用户账号管理`

### 第 1 阶段页面范围

- 后台主布局
- Dashboard 总览页
- 记忆管理页（重构接入）
- 知识库管理骨架页
- 系统设置骨架页
- 用户账号管理基础页

### 页面交互规范

- 左侧为主导航，右侧为内容区
- 顶栏展示当前模块、面包屑、用户菜单和全局状态
- 列表页统一提供筛选栏、搜索框、状态标签、空状态和分页区
- 详情编辑统一采用抽屉或弹窗，不在页面内随意散落编辑表单
- 高风险操作统一使用确认弹窗

## 6. UI/UX 设计规范

后台整体设计采用现代管理台风格，不沿用当前前台的“轻工具页拼接式布局”。

### 视觉方向

- 深色侧边导航 + 浅色主内容区
- 卡片化信息分组
- 明确的标题层级、分区边界、状态色和操作反馈
- 保持专业、克制、稳定，不做偏娱乐化视觉

### 统一设计元素

- 统一色板：主色、警告色、危险色、成功色、弱化灰阶
- 统一字体体系：标题、正文、数据数值、标签分层明确
- 统一组件：表格、筛选条、状态徽标、按钮、空状态、弹窗、表单控件
- 统一反馈：加载、成功、错误、禁用、危险操作确认

### 响应式策略

- 桌面端：完整左侧导航 + 多列内容区
- 平板端：压缩导航宽度，保留侧边栏
- 移动端：导航折叠为抽屉，内容区改为单列堆叠

## 7. 前台与后台的边界

### 设置模块

用户前台不再承载完整设置能力。后台成为系统设置的主入口。

第 1 阶段策略：

- 前台现有设置页面可保留为只读或精简版
- 后台提供正式设置模块入口和布局骨架
- 配置修改能力后续收敛到后台

### 知识库模块

用户前台保留“可见文件”的只读访问能力，但不允许编辑。

后台负责：

- 文件可见范围控制
- 草稿 / 已发布状态管理
- 文件元数据维护
- 发布和隐藏操作

## 8. 知识库可见性与版本模型

用户已确认知识库采用“按角色控制可见范围”，版本控制采用“草稿 / 已发布”双态。

### 可见范围模型

每个知识库文件需要增加角色可见范围信息，例如：

- 仅后台可见
- `user` 可见
- `operator` 可见
- `admin` 可见

实现上建议采用角色集合字段，支持后续扩展。

### 版本控制模型

每个文件至少维护：

- 当前草稿内容
- 当前已发布内容
- 发布时间
- 发布人
- 草稿更新时间
- 草稿编辑人

第 1 阶段先补齐数据模型和后台入口，不要求一次性做完整版本历史浏览器。

## 9. 用户账号管理范围

第 1 阶段的用户账号管理仅覆盖“后台管理员所需的基础治理能力”：

- 用户列表
- 用户搜索与筛选
- 查看用户详情
- 账号启用 / 禁用
- 角色修改
- 管理员重置密码

不在第 1 阶段范围内：

- 邮件验证码或短信找回
- 双因子认证
- 审批流
- 组织架构

## 10. 后端架构设计

后台 API 从当前单一的 `memory` 管理接口扩展为统一后台接口集合。

### 推荐目录结构

- `backend/app/api/admin/dashboard.py`
- `backend/app/api/admin/memory.py`
- `backend/app/api/admin/knowledge.py`
- `backend/app/api/admin/settings.py`
- `backend/app/api/admin/users.py`

服务层按模块拆分：

- `backend/app/services/memory_admin_service.py`
- `backend/app/services/knowledge_admin_service.py`
- `backend/app/services/settings_admin_service.py`
- `backend/app/services/user_admin_service.py`
- `backend/app/services/permission_service.py`
- `backend/app/services/audit_log_service.py`

### 设计原则

- 路由层只做参数校验、权限门面和响应组装
- 业务逻辑放在 service 层
- 权限判断抽到统一 service，不在每个模块手写散乱条件
- 审计日志统一写入，避免模块各自实现

## 11. 数据模型调整

### 用户表扩展

现有用户表需要至少增加：

- `role`
- `status`
- `last_login_at`
- `password_updated_at`
- `updated_at`

### 审计日志表

建议新增审计日志表，记录：

- 操作人 ID
- 操作模块
- 操作类型
- 目标对象类型
- 目标对象 ID
- 操作摘要
- 结果状态
- 时间戳
- IP / 请求上下文（如可获取）

### 知识库元数据

建议为知识库文件维护结构化元数据：

- `visibility_scope`
- `is_frontend_visible`
- `draft_status`
- `published_status`
- `published_at`
- `published_by`

## 12. 安全要求

### 认证与鉴权

- 后台沿用现有 token 机制
- token 解析后必须校验角色
- 后台路由必须显式声明所需角色

### 风险控制

- 删除、发布、重置密码、角色变更必须记录审计日志
- 高风险操作必须二次确认
- 后台返回数据时不暴露敏感密码字段或密钥原文

### 数据校验

- 所有后台请求使用明确 schema 校验
- 文件名、用户 ID、配置项和角色值必须做白名单校验

## 13. 日志与可观测性

系统需补齐后台治理所需日志能力：

- 认证失败日志
- 后台访问拒绝日志
- 高风险操作审计日志
- 模块级错误日志
- 配置变更日志

第 1 阶段不要求接入外部监控平台，但必须在代码结构上保留后续接入空间。

## 14. 测试策略

第 1 阶段至少覆盖：

### 单元测试

- 角色权限判断
- 用户管理 service
- 后台 API 参数校验
- 知识库可见范围判断
- 审计日志写入

### 集成测试

- 管理员登录并访问后台模块
- 普通用户访问后台被拒绝
- 管理员修改用户角色成功
- 管理员更新记忆数据成功
- 管理员控制知识库前台可见性成功

### 前端验证

- 后台主框架渲染正确
- 角色不同导致菜单不同
- 移动端导航可用
- 高风险操作弹窗与错误提示可用

## 15. 第 1 阶段验收标准

满足以下条件视为第 1 阶段完成：

- 已有独立后台框架，具备统一导航与统一设计语言
- 后台登录复用现有账号体系，并按角色控制访问
- `super_admin / admin / operator / user` 角色模型已落地
- 记忆管理模块已接入后台主框架
- 知识库管理和系统设置已迁入后台导航体系并具备骨架页
- 用户账号管理基础能力可用
- 后台接口具备统一权限校验
- 高风险操作具备审计日志
- 基础单元测试和集成测试可运行

## 16. 不在本阶段范围内

为了控制交付风险，以下内容不纳入第 1 阶段：

- 完整知识库高级版本历史浏览
- 多组织、多租户
- 邮件 / 短信找回密码
- 双因子认证
- 审批流
- 细粒度到字段级的权限编排器

## 17. 实施顺序建议

第 1 阶段内部建议按以下顺序推进：

1. 用户表与角色模型扩展
2. 后台权限判断与统一鉴权
3. 后台主框架与页面骨架
4. 用户账号管理基础页
5. 记忆管理重构接入
6. 知识库与设置后台入口接入
7. 审计日志与测试补齐
